# 관계형 DB 역량 어필 문장 / 개선 과제 정리

## 면접 30초 답변 버전

이 프로젝트에서는 배우, 영화, 필모그래피, 미디어 인코딩 작업을 관계형 데이터베이스 기준으로 분리 설계하고, 각 엔티티의 관계와 상태를 일관되게 관리하려고 했습니다. 특히 `media_job` 테이블로 비동기 인코딩 상태를 추적하고, 상태 전이 규칙을 둬 데이터 정합성을 유지하도록 했습니다. 추가로 인덱스 설계, 락 전략, 조회 최적화를 붙이면 관계형 DB 활용 역량을 더 강하게 보여줄 수 있습니다.

---

## 1. 포트폴리오에 바로 쓸 수 있는 어필 문장 3개

### 1-1. 관계 설계 중심

배우 프로필, 영화, 필모그래피, 트레일러/썸네일/본편 미디어는 단순 문서형 데이터가 아니라 서로 연결된 관계를 가지기 때문에, 관계형 데이터베이스 기준으로 엔티티를 분리하고 연결 구조를 설계했습니다. 특히 필모그래피는 단순 영화 목록이 아니라 배우-영화 사이의 역할, 배역명, 공개 여부 같은 속성을 함께 관리해야 했기 때문에 관계와 속성을 함께 다루는 모델링 경험을 쌓을 수 있었습니다.

### 1-2. 상태 관리와 정합성 중심

비동기 인코딩 파이프라인은 Kafka 메시지로만 끝내지 않고 `media_job` 테이블을 별도로 두어 작업 상태를 추적했습니다. `job_id`, `movie_id`, `job_type`, `status`, `target_key`, `started_at`, `completed_at`, `failure_reason` 등을 컬럼으로 나눠 저장했고, `REQUESTED -> PROCESSING -> DONE|FAILED` 같은 상태 전이 규칙을 애플리케이션 레벨에서 제한해 비동기 처리 중에도 데이터 정합성을 유지하도록 설계했습니다.

### 1-3. 조회 구조와 JPA 활용 중심

프로필 상세와 필모그래피 조회는 여러 테이블에 흩어진 데이터를 하나의 응답으로 조합해야 했기 때문에, 어떤 데이터는 연관 조회로 가져오고 어떤 데이터는 분리 조회할지 기준을 세워 설계했습니다. 이 과정에서 JPA 연관관계, 읽기 전용 트랜잭션, N+1 문제 가능성을 함께 점검했고, 관계형 데이터 모델 위에서 조회 구조를 어떻게 설계해야 하는지 경험할 수 있었습니다.

## 2. 현재 프로젝트에 추가하면 좋은 개선 과제 5개

### 2-1. `media_job` 조회 패턴에 맞는 인덱스 추가

목표:
- 작업 상태 조회와 운영성 조회를 빠르게 만들기

추천 포인트:
- `media_job(job_id)`는 PK로 유지
- `media_job(movie_id, status)`
- 필요 시 `media_job(status, created_at)`

어필 포인트:
- 단순히 인덱스를 추가한 것이 아니라 실제 조회 패턴을 기준으로 설계했다는 점을 말할 수 있음
- `EXPLAIN`과 응답 시간을 비교하면 수치 기반 사례가 됨

예시 문장:
`인코딩 상태 조회와 운영성 확인 쿼리의 사용 패턴을 기준으로 media_job에 복합 인덱스를 설계하고, EXPLAIN으로 실행 계획을 비교해 풀스캔을 인덱스 탐색으로 바꿨습니다.`

### 2-2. 필모그래피 정렬/재정렬에 대한 정합성 강화

목표:
- 드래그 정렬 저장 시 순서 꼬임이나 동시 수정 문제 방지

추천 포인트:
- 배우별 필모그래피 순서를 나타내는 정렬 컬럼에 대해 일관된 update 전략 정리
- `person_id + sort_order` 조합 관리
- 충돌 가능성이 있으면 낙관적 락 또는 정렬 재계산 로직 검토

어필 포인트:
- 관계형 DB에서 “순서 데이터”를 안전하게 다뤘다는 점을 보여줄 수 있음
- 단순 CRUD가 아니라 동시성/정합성 문제를 다뤘다는 인상을 줄 수 있음

예시 문장:
`필모그래피 순서 저장은 단순 리스트 덮어쓰기가 아니라, 배우 단위 정렬값의 일관성을 유지하도록 설계해 순서 꼬임과 동시 수정 가능성을 줄였습니다.`

### 2-3. `refresh token` 재발급에 대한 트랜잭션/락 적용

목표:
- 동시에 여러 재발급 요청이 들어올 때 중복 발급이나 상태 꼬임 방지

추천 포인트:
- refresh token row에 비관적 락 적용 검토
- 재발급 시 이전 토큰 무효화와 새 토큰 저장을 하나의 트랜잭션으로 묶기

어필 포인트:
- 관계형 DB를 단순 저장소가 아니라 동시성 제어 도구로 활용한 사례가 됨
- “트랜잭션과 락을 실제 어디에 왜 썼는지” 설명 가능

예시 문장:
`토큰 재발급은 동시 요청 시 중복 발급 위험이 있어, refresh token row를 기준으로 트랜잭션과 비관적 락을 적용하는 방향으로 설계했습니다.`

### 2-4. 프로필/필모그래피 상세 조회의 N+1 제거

목표:
- 상세 페이지 진입 시 쿼리 수와 응답 시간을 줄이기

추천 포인트:
- 현재 배우 상세, 필모그래피, 미디어 URL 조합 조회에서 쿼리 수 측정
- fetch join, projection, DTO 조회로 분리 최적화

어필 포인트:
- JPA를 단순히 엔티티 저장용으로만 쓰지 않고, 조회 최적화 관점에서 다뤘다는 점을 보여줄 수 있음
- 로그 기준으로 “쿼리 수 몇 개에서 몇 개로 감소” 같은 수치를 만들기 좋음

예시 문장:
`배우 상세 페이지의 연관 데이터 조회에서 N+1 가능성을 점검하고, fetch join 또는 DTO projection으로 쿼리 수를 줄이는 최적화를 진행했습니다.`

### 2-5. DB 제약조건을 활용한 무결성 강화

목표:
- 애플리케이션 코드뿐 아니라 DB 레벨에서도 잘못된 데이터 입력 방지

추천 포인트:
- username/email unique constraint 점검
- `media_job.job_id` 중복 방지
- 필요 시 필모그래피 매핑에 unique/foreign key 제약 강화

어필 포인트:
- 관계형 DB의 핵심인 무결성 보장을 이해하고 있다는 점을 보여줄 수 있음
- “서비스 로직에서만 막지 않고 DB에서도 막았다”는 식으로 설명 가능

예시 문장:
`애플리케이션 검증만으로 끝내지 않고 unique constraint와 foreign key를 통해 DB 레벨에서도 무결성을 보장하도록 설계했습니다.`

## 3. 실제 포트폴리오에 쓰기 좋은 정리 방식

아래 형식으로 쓰면 읽는 사람이 DB 역량을 빠르게 파악하기 쉽다.

### 형식 예시

- 문제:
  비동기 미디어 작업 상태를 메모리나 메시지 큐에만 의존하면, 작업 이력 추적과 상태 정합성 관리가 어려웠다.
- 해결:
  `media_job` 테이블을 별도로 두고 작업 상태와 결과 경로를 컬럼 단위로 관리했으며, 상태 전이 규칙을 제한해 정합성을 유지했다.
- 결과:
  인코딩 진행 상태를 DB에서 일관되게 조회할 수 있게 되었고, API 서버와 워커 서버의 역할을 명확히 분리할 수 있었다.

### 작성 팁

- “테이블을 만들었다”보다 “왜 분리했는지”를 먼저 쓴다.
- “JPA를 썼다”보다 “어떤 조회 문제를 다뤘는지”를 쓴다.
- 가능하면 `쿼리 수`, `응답 시간`, `EXPLAIN`, `락 충돌 방지` 같은 근거를 붙인다.
- 아직 수치가 없다면 “이 패턴을 기준으로 측정하고 개선하려 했다”까지는 말할 수 있다.

## 4. 지금 당장 우선순위로 추천하는 DB 개선 3개

1. `media_job` 인덱스 설계 및 `EXPLAIN` 비교 자료 만들기
2. 배우 상세/필모그래피 조회 쿼리 수 측정 후 N+1 제거하기
3. `refresh token` 또는 필모그래피 정렬 저장에 트랜잭션/락 전략 적용하기

이 3개만 해도 “관계형 DB를 제대로 이해하고 다뤘다”는 인상을 훨씬 강하게 줄 수 있다.
