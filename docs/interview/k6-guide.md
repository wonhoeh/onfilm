# K6 부하테스트 가이드

이 문서는 `onfilm` 프로젝트에서 `k6`로 어디를 측정하면 좋은지 정리한 문서다.

## 면접 30초 답변 버전

“이 프로젝트에서 k6를 돌린다면 단순 RPS보다 병목이 실제로 생길 수 있는 경로를 나눠서 보겠습니다. 공개 조회 API는 `p95`와 SQL 개수를 같이 보고, 인증 API는 refresh token 재발급의 동시성 정합성을 확인하고, 업로드 쪽은 presigned URL 발급과 media job 생성 API가 높은 동시성에서도 가벼운지 보겠습니다. 또 reorder나 upsert는 응답 시간보다 최종 데이터가 꼬이지 않는지까지 함께 검증하는 게 핵심입니다.”

핵심은 단순 RPS보다 아래를 같이 보는 것이다.

- `p95`, `p99` 응답 시간
- 에러율
- 동시 요청 시 데이터 정합성
- SQL 개수와 느린 쿼리 로그
- CPU, 메모리, DB connection pool 사용량

## 1. 우선순위가 높은 대상

### 1-1. 공개 조회 API

대상:

- 프로필 조회
- publicId 조회
- 필모그래피 조회

이유:

- 읽기 트래픽이 가장 많이 몰릴 가능성이 높다.
- 필모그래피 조회는 JPA 조회 최적화가 실제로 효과가 있는지 확인하기 좋다.

보는 지표:

- `http_req_duration`
- `http_req_failed`
- SQL 개수
- DB slow query

## 1-2. 인증 API

대상:

- 로그인
- 토큰 재발급
- 로그아웃
- 회원가입

이유:

- 트랜잭션 정합성 검증이 중요하다.
- 특히 `refresh token` 재발급은 동시성 충돌 테스트가 필요하다.

보는 지표:

- 응답 시간
- 실패율
- 동일 토큰 동시 요청 시 중복 발급 여부

## 1-3. 업로드 관련 API

대상:

- presigned URL 발급
- 업로드 완료 후 media job 생성
- media job polling

이유:

- 서버가 파일을 직접 받지 않는 구조가 얼마나 가벼운지 검증 가능하다.
- Kafka job 저장/조회가 몰릴 때 API 응답성을 확인할 수 있다.

보는 지표:

- API 응답 시간
- DB write latency
- Kafka producer 전송 지연

## 1-4. 수정 API

대상:

- 필모그래피 upsert
- 갤러리 reorder
- storyboard scene reorder / update

이유:

- 같은 리소스에 대한 동시 수정 충돌 가능성이 높다.

보는 지표:

- 응답 시간
- 실패율
- 최종 데이터 정합성

## 2. 추천 시나리오

## 2-1. 공개 조회 부하

목표:

- 조회 API 최대 처리량 확인
- `p95` 상승 시점 확인

확인 포인트:

- 읽기 전용 트랜잭션 구간에서 DB 병목 발생 여부
- 필모그래피 조회의 SQL 수가 예상 범위 내인지

## 2-2. refresh token 동시 재발급

목표:

- 동일 refresh token으로 동시 요청 시 하나만 정상 처리되는지 검증

확인 포인트:

- 새 refresh token이 중복 발급되는지
- 기존 토큰이 정확히 revoke 되는지

이 시나리오는 락 전략 개선 전후 비교에 가장 유용하다.

## 2-3. 회원가입 중복 경쟁

목표:

- 같은 이메일/username으로 동시 요청 시 정합성 검증

기대 결과:

- 성공은 1건
- 나머지는 `409` 또는 unique violation 기반 실패

## 2-4. reorder 경쟁

목표:

- 같은 갤러리/스토리보드에 서로 다른 reorder 요청 동시 발생 시 최종 상태 확인

기대 결과:

- 낙관적 락 도입 전: 마지막 쓰기 승리 가능성 확인
- 낙관적 락 도입 후: 충돌 시 `409` 처리

## 2-5. 업로드 요청 폭주

목표:

- presigned URL 발급과 media job 생성 API가 높은 동시성에서도 가벼운지 확인

기대 결과:

- 파일 본문이 서버를 통과하지 않기 때문에 API 자체는 비교적 안정적이어야 함

## 3. 테스트 전 준비

- Actuator metrics 활성화
- HikariCP metrics 확인 가능하게 설정
- SQL 로그 또는 slow query log 켜기
- 고정 테스트 계정 준비
- 고정 refresh token 준비
- 충분한 샘플 데이터 준비

## 4. 면접 답변 연결 문장

“k6로 읽기 API와 인증 API를 분리해서 봤습니다. 읽기 API는 `p95`와 SQL 수를 같이 확인했고, 인증 API는 단순 응답 시간보다 동시성 정합성을 더 중요하게 봤습니다. 특히 refresh token 재발급은 같은 토큰으로 동시 요청이 들어왔을 때 중복 발급이 생기지 않는지를 별도 시나리오로 검증하는 게 핵심이라고 판단했습니다.”
