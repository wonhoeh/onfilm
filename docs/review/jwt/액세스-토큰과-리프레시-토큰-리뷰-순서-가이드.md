# Access Token / Refresh Token 코드 리뷰 순서 가이드

## 개요

인증 코드는 관련 클래스가 여러 군데 흩어져 있어서 한 번에 보면 헷갈리기 쉽다.  
가장 좋은 방법은 `로그인 -> access token 발급 -> 요청 인증 -> access token 만료 -> refresh -> 재발급 -> logout` 흐름으로 따라가는 것이다.

즉 “파일 이름 순서”가 아니라 “기능 흐름 순서”로 보는 게 중요하다.

---

## 1. 가장 먼저 볼 전체 흐름

이 프로젝트의 인증/세션 흐름은 대략 아래처럼 이해하면 된다.

1. 로그인 요청이 들어온다.
2. 서버가 사용자 인증에 성공하면 access token을 발급한다.
3. 동시에 refresh token을 발급하고 DB에 저장한다.
4. 이후 요청마다 access token을 검증한다.
5. access token이 만료되면 refresh token으로 새 access token을 재발급한다.
6. logout 또는 rotation 시 refresh token을 폐기한다.

즉:

- access token = 매 요청 인증용
- refresh token = 재발급 전용

이라는 역할 분리부터 먼저 잡고 시작하면 된다.

이 파일에서 먼저 “인증 기능의 외부 인터페이스”를 잡는다.

## 2-2. [`AuthService.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/auth/service/AuthService.java)

두 번째로 보는 이유:

- 실제 인증 흐름의 중심 서비스이기 때문

리뷰 포인트:

- 로그인 성공 시 access token은 어디서 만드는가

- refresh 요청이 들어오면 어떤 순서로 재발급하는가

## 2-3. [`JwtProvider.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/common/config/JwtProvider.java)

세 번째로 보는 이유:

리뷰 포인트:

- access token에 어떤 클레임이 들어가는가

이 파일에서 “access token / refresh token의 기술적 차이”를 본다.

## 3. 리뷰할 때 따라가면 좋은 질문

아래 질문 순서로 보면 훨씬 덜 헷갈린다.

### 3-1. 로그인

- 로그인 성공 시 access token은 어디서 발급되는가?
- refresh token은 어디서 발급되는가?
- 둘은 어디에 담겨서 내려가는가?

### 3-2. 요청 인증

- 클라이언트는 access token을 어떻게 보낸다고 가정하는가?
- 서버는 access token을 어디서 읽는가?
- validate 후 SecurityContext는 어떻게 채워지는가?

### 3-3. refresh

- access token이 만료되면 어떤 API를 호출하는가?
- refresh token은 DB에서 어떤 기준으로 검증하는가?
- rotation은 어떻게 구현되어 있는가?

### 3-4. logout / revoke

- logout 시 무엇이 무효화되는가?
- access token은 즉시 끊을 수 있는가?
- refresh token은 어떻게 revoke되는가?

## 5. 왜 이 순서가 좋은가

이 순서가 좋은 이유는 “바깥에서 안으로” 들어가기 때문이다.

- 먼저 API 입구를 본다.
- 그 다음 서비스 흐름을 본다.
- 그 다음 JWT 생성/검증을 본다.
- 그 다음 필터에서 실제 인증 연결을 본다.
- 마지막에 refresh token의 상태 관리와 설정값을 본다.

즉 처음부터 엔티티나 설정 파일부터 보면 추상적으로 느껴지는데,  
Controller -> Service -> Provider/Filter -> Token Service/Entity 순으로 보면 실제 동작이 먼저 잡힌다.
