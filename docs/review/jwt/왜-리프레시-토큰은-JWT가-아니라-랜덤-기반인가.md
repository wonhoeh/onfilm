# 왜 Refresh Token은 JWT가 아니라 랜덤 기반으로 두는가

## 1. 결론 요약

이 프로젝트에서 refresh token을 JWT가 아니라 랜덤 문자열(opaque token)로 만든 이유는, refresh token이 access token보다 수명이 길고 보안 민감도가 높기 때문에 서버가 DB를 기준으로 더 강하게 통제할 수 있는 구조가 필요하기 때문이다.

즉 역할을 나누면 이렇게 볼 수 있다.

- access token
  - 짧은 수명
  - 요청마다 자주 사용
  - 빠른 stateless 검증이 중요
- refresh token
  - 긴 수명
  - 재발급 시에만 사용
  - 강한 통제와 무효화가 중요

그래서 access token은 JWT, refresh token은 DB 기반 랜덤 토큰으로 나누는 것이 실무적으로 자연스럽다.

## 2. 현재 프로젝트는 어떻게 구현되어 있는가

현재 프로젝트에서는 refresh token을 JWT로 만들지 않는다.

관련 코드:

- [`RefreshToken.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/token/entity/RefreshToken.java)

구조를 보면:

- 서버는 랜덤 문자열 refresh token을 생성
- DB에는 그 원문이 아니라 해시를 저장
- 만료 시간, 폐기 시간, 마지막 사용 시간도 함께 저장

즉 refresh token은 “자체적으로 의미를 가진 JWT”가 아니라, “서버가 식별 가능한 랜덤 세션 키”에 가깝다.

---

## 3. 왜 단순 서명 검증보다 DB 직접 확인이 더 안전한가

### 3-1. JWT는 서명이 유효하면 기본적으로 신뢰하게 된다

JWT 기반 refresh token은 보통 아래를 확인한다.

- 서명이 올바른가
- 만료 시간이 지나지 않았는가

문제는 이 구조에서는 “서명과 만료만 유효하면” 서버 입장에서 토큰을 유효하다고 보기 쉽다는 점이다.

즉 다음 질문에 바로 답하기 어렵다.

- 사용자가 로그아웃한 토큰인가?
- 이미 재발급에 사용된 토큰인가?
- 탈취 의심으로 폐기한 토큰인가?
- 같은 계정의 다른 세션인가?

JWT 자체는 이 상태를 실시간으로 반영하지 못한다.

### 3-2. DB 기반이면 토큰 상태를 서버가 즉시 판단할 수 있다

DB 기반 refresh token은 토큰이 들어올 때마다 서버가 직접 상태를 확인할 수 있다.

예를 들면:

- 해시가 DB에 존재하는가
- `revokedAt`이 비어 있는가
- `expiresAt`이 지나지 않았는가
- 마지막 사용 시간이 비정상적이지 않은가

즉 “이 토큰이 수학적으로 유효한가?”가 아니라,  
“현재 서비스 정책상 이 토큰을 계속 인정할 것인가?”를 판단할 수 있다.

### 3-3. 왜 이게 더 안전한가

refresh token은 수명이 길어서 탈취되면 피해가 크다.  
그런데 JWT만 믿으면 토큰이 탈취된 뒤에도 만료 전까지 계속 재발급에 쓰일 수 있다.

반면 DB 기반이면:

- 서버가 특정 토큰을 바로 폐기할 수 있고
- 이미 사용된 토큰을 다시 막을 수 있고
- 이상 징후가 보이면 세션 단위로 차단할 수 있다

즉 refresh token에서는 “stateless 편의성”보다 “상태 기반 통제력”이 더 중요하기 때문에 DB 직접 확인이 더 안전하다고 보는 것이다.

---

## 4. 왜 강제 무효화가 쉬운가

### 4-1. JWT refresh token은 발급 후 되돌리기 어렵다

JWT는 한 번 발급되면 그 자체 안에 만료 시간과 사용자 정보가 들어 있다.  
그래서 서버가 “이 토큰은 지금부터 더 이상 쓰면 안 된다”라고 판단해도, 별도 블랙리스트 저장소가 없으면 즉시 막기 어렵다.

즉 JWT refresh token을 강제 무효화하려면 결국:

- 블랙리스트 저장소
- 토큰 ID(`jti`) 저장
- 요청마다 블랙리스트 조회

같은 추가 상태 관리가 필요하다.

그렇게 되면 결국 “stateless JWT” 장점이 많이 사라진다.

### 4-2. DB 기반 refresh token은 row 하나 업데이트하면 끝난다

현재 구조처럼 refresh token이 DB row로 관리되면 강제 무효화는 단순하다.

예:

- `revokedAt = now()`

이렇게만 바꾸면 그 다음부터는 같은 토큰이 들어와도 재발급을 막을 수 있다.

### 4-3. 왜 이게 중요한가

다음 상황에서 즉시 무효화가 중요하다.

- 로그아웃
- 비밀번호 변경
- 계정 탈취 의심
- 특정 기기 세션 종료
- refresh token rotation 중 재사용 탐지

즉 refresh token은 “나중에 자연 만료될 때까지 두기엔 위험한” 장기 자격 증명이기 때문에, 서버가 즉시 끊을 수 있어야 한다.

---

## 5. 왜 rotation에 유리한가

### 5-1. rotation이란 무엇인가

refresh token rotation은 재발급 시:

1. 기존 refresh token을 폐기하고
2. 새 refresh token을 발급하고
3. 이후에는 새 토큰만 인정하는 방식이다

이 방식은 탈취 피해를 줄이는 데 유리하다.

### 5-2. 왜 DB 기반이 rotation에 자연스러운가

DB 기반 구조에서는 refresh 요청이 들어오면 다음을 한 트랜잭션 안에서 처리하기 쉽다.

- 기존 token row 조회
- 유효성 확인
- 기존 token revoke
- 새 token row 생성

즉 토큰의 “이전 상태”와 “새 상태”를 서버가 직접 연결해서 관리할 수 있다.

### 5-3. 왜 JWT refresh token은 불편한가

JWT refresh token으로도 rotation은 가능하지만, 결국 아래가 필요해진다.

- 이전 토큰을 더 이상 못 쓰게 할 저장소
- 이미 사용된 토큰인지 추적할 로직
- 재사용 공격 탐지 로직

즉 회전 자체는 가능하지만, “JWT만으로”는 해결이 안 되고 결국 상태 저장소가 따라온다.

그럴 바엔 처음부터 refresh token을 DB 중심으로 관리하는 편이 더 단순하고 명확하다.

### 5-4. 왜 보안상 유리한가

탈취자가 예전 refresh token을 가지고 있다가 나중에 다시 쓰는 경우를 생각해보면:

- DB 기반 rotation이면 “이미 폐기된 토큰 재사용”을 쉽게 탐지할 수 있다
- 그 순간 해당 세션 전체를 차단하는 정책도 적용할 수 있다

즉 rotation은 단순 만료보다 훨씬 강한 방어 수단이고, DB 기반 구조가 이를 구현하기 쉽다.

---

## 6. 왜 토큰 내용이 안 보이는 것이 장점인가

### 6-1. JWT payload는 암호화가 아니라 인코딩이다

JWT는 보통 서명되어 있지만, payload 자체는 누구나 Base64 디코딩으로 읽을 수 있다.

즉 JWT refresh token을 클라이언트가 들고 있으면, 그 안의 정보는 쉽게 확인할 수 있다.

물론 민감한 정보를 안 넣으면 큰 문제는 아니지만, “토큰 자체가 자기 설명적인 구조”라는 점은 남는다.

### 6-2. 랜덤 토큰은 읽어도 아무 의미가 없다

랜덤 기반 refresh token은 단순한 무작위 문자열이라서, 클라이언트가 보더라도 내부 의미를 알 수 없다.

예:

```text
Q2f8k2Nn1c...랜덤 문자열...
```

이 값만 보고는:

- 누구 토큰인지
- 언제 만료되는지
- 어떤 발급자인지

같은 정보를 전혀 알 수 없다.

### 6-3. 왜 이게 장점인가

이 구조는 최소한의 정보 노출 원칙에 맞다.

즉 refresh token은:

- 자주 읽히거나 디버깅될 필요가 없고
- 외부에 의미가 드러날 필요도 없기 때문에

랜덤 opaque token이 더 목적에 맞다.

---

## 7. 왜 refresh token은 stateful이어도 부담이 적은가

access token은 모든 요청마다 붙기 때문에 매 요청마다 DB 조회를 하면 부담이 크다.  
그래서 access token은 stateless JWT가 잘 맞는다.

반면 refresh token은:

- 로그인 후 자주 쓰지 않고
- access token이 만료될 때만 사용되고
- 보통 `/auth/refresh` 같은 제한된 시점에만 사용된다

즉 요청 빈도가 낮기 때문에 DB 확인 비용을 감수해도 괜찮다.

그래서 refresh token은:

- 성능보다 보안/통제
- stateless보다 stateful 관리

가 더 잘 맞는다.

---

## 8. 정리하면 왜 랜덤 기반이 더 적합한가

refresh token을 랜덤 기반으로 두면 다음이 가능해진다.

1. 서버가 DB에서 상태를 직접 확인할 수 있다.
2. 로그아웃이나 이상 징후 발생 시 즉시 무효화할 수 있다.
3. rotation을 자연스럽게 구현할 수 있다.
4. 토큰 자체에 의미 있는 정보가 드러나지 않는다.
5. 요청 빈도가 낮아서 DB 조회 부담이 크지 않다.

즉 refresh token은 “빠른 검증”보다 “강한 통제”가 중요한 토큰이라서, JWT보다 DB 기반 랜덤 토큰이 더 적합하다고 볼 수 있다.

---

## 9. 면접에서 설명하는 방식

아래처럼 설명하면 좋다.

`refresh token은 access token보다 수명이 길고 탈취 시 위험이 커서, 단순히 서명과 만료 시간만 확인하는 JWT 구조보다 DB에서 상태를 직접 통제할 수 있는 랜덤 토큰 방식이 더 적절하다고 판단했습니다. 이 방식이면 로그아웃이나 탈취 의심 상황에서 즉시 revoke할 수 있고, refresh token rotation도 기존 토큰 폐기와 새 토큰 발급을 트랜잭션 안에서 자연스럽게 처리할 수 있습니다.`

---

## 10. 한 줄 정리

access token은 빠르게 검증해야 해서 JWT가 잘 맞고, refresh token은 오래 살아서 강한 통제가 필요하므로 DB 기반 랜덤 토큰이 더 잘 맞는다.
