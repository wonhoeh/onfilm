# X-CSRF-TOKEN 정리

## 1. X-CSRF-TOKEN이란?

`X-CSRF-TOKEN`은 CSRF 공격을 막기 위해 브라우저가 요청 헤더에 담아 보내는 보안용 값이다.

쉽게 말하면:

- 브라우저는 쿠키를 자동으로 보낸다
- 그래서 쿠키 기반 인증만 쓰면 공격자가 사용자의 브라우저를 악용해 요청을 보내게 만들 수 있다
- 이를 막기 위해 “쿠키만 자동으로 붙은 요청”이 아니라, “프론트 화면이 직접 만든 정상 요청”이라는 추가 증거가 필요하다

그 추가 증거가 바로 `X-CSRF-TOKEN` 헤더다.

---

## 2. 왜 필요한가

쿠키 기반 인증에서는 브라우저가 요청할 때 쿠키를 자동으로 붙여서 보낸다.

이건 편리하지만 동시에 위험도 있다.

예를 들어:

1. 사용자가 로그인된 상태다
2. 악성 사이트에 접속한다
3. 그 사이트가 사용자의 브라우저를 이용해 우리 서버로 `POST`, `PUT`, `DELETE` 요청을 보내게 만든다
4. 브라우저는 로그인 쿠키를 자동으로 붙인다

이게 바로 CSRF(Cross-Site Request Forgery)다.

즉 서버 입장에서는:

- 쿠키가 붙어 있다고 해서
- 그 요청이 진짜 우리 화면에서 발생한 요청인지

확신할 수 없다.

그래서 쿠키 외에 별도 값을 함께 확인하는 구조가 필요하다.

---

## 3. 현재 프로젝트에서는 어떻게 동작하는가

현재 프로젝트는 CSRF 방어를 위해 아래 조합을 사용한다.

- CSRF 쿠키
  - `XSRF-TOKEN`
- 요청 헤더
  - `X-CSRF-TOKEN`

흐름은 이렇다.

1. 서버가 로그인/refresh 시 CSRF 쿠키를 내려준다
2. 프론트 JS가 `XSRF-TOKEN` 쿠키를 읽는다
3. 상태 변경 요청(`POST`, `PUT`, `PATCH`, `DELETE`)을 보낼 때
4. 같은 값을 `X-CSRF-TOKEN` 헤더에 넣어 보낸다
5. 서버는 쿠키 값과 헤더 값이 모두 존재하고 서로 같은지 확인한다

즉 서버는:

- 쿠키도 있고
- 헤더도 있고
- 두 값이 같을 때만

정상 요청으로 본다.

---

## 4. 왜 쿠키와 헤더 둘 다 봐야 하는가

핵심은 “공격자는 브라우저의 자동 쿠키 전송은 유도할 수 있어도, 우리 프론트처럼 정확한 값을 읽어서 커스텀 헤더로 넣기는 어렵다”는 점이다.

즉:

- 쿠키만 검사하면 공격자가 악성 요청을 보내도 쿠키가 자동 전송될 수 있다
- 헤더까지 같이 검사하면, 정상 프론트가 직접 만든 요청만 통과시키기 쉬워진다

그래서 서버는 보통:

- `XSRF-TOKEN` 쿠키 값
- `X-CSRF-TOKEN` 헤더 값

두 개를 비교한다.

이 방식을 흔히 `double submit cookie` 패턴이라고 부른다.

---

## 5. 현재 프로젝트 코드에서는 어디서 검사하는가

현재 프로젝트에서는 [`CsrfProtectionFilter.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/common/config/CsrfProtectionFilter.java)에서 검사한다.

핵심 확인 항목은 다음과 같다.

- 요청 메서드가 안전한 메서드인지
- `Origin` 또는 `Referer`가 same-origin인지
- `XSRF-TOKEN` 쿠키가 있는지
- `X-CSRF-TOKEN` 헤더가 있는지
- 쿠키 값과 헤더 값이 같은지

즉 이 프로젝트의 `X-CSRF-TOKEN`은 단순한 커스텀 헤더가 아니라, 커스텀 CSRF 필터가 직접 검증하는 보안 값이다.

---

## 6. 왜 GET 요청은 보통 제외하는가

CSRF는 보통 서버 상태를 바꾸는 요청에서 문제가 된다.

즉 주로 위험한 요청은:

- `POST`
- `PUT`
- `PATCH`
- `DELETE`

같은 변경 요청이다.

반면:

- `GET`
- `HEAD`
- `OPTIONS`

같은 조회성 요청은 보통 상태를 바꾸지 않으므로 CSRF 검사 대상에서 제외하는 경우가 많다.

현재 프로젝트도 `CsrfProtectionFilter`에서 안전한 메서드는 기본적으로 스킵한다.

---

## 7. 현재 프로젝트에서 쿠키와 헤더 이름은 무엇인가

현재 기본값 기준으로:

- 쿠키 이름: `XSRF-TOKEN`
- 헤더 이름: `X-CSRF-TOKEN`

이 조합을 사용한다.

즉 프론트는 쿠키 `XSRF-TOKEN`을 읽어서 요청 헤더 `X-CSRF-TOKEN`에 넣어야 한다.

---

## 8. 왜 `HttpOnly(false)`가 필요한가

CSRF 쿠키는 프론트 JS가 읽어서 헤더에 넣어야 하므로, `HttpOnly(true)`로 만들면 안 된다.

왜냐면:

- `HttpOnly(true)` 쿠키는 JS가 읽을 수 없기 때문이다

즉:

- access token / refresh token은 `HttpOnly(true)`가 적합
- CSRF 쿠키는 JS가 읽어야 하므로 `HttpOnly(false)`가 필요

이 차이를 이해하는 것이 중요하다.

---

## 9. 현재 프로젝트에서 왜 필요한가

현재 프로젝트는 쿠키 기반으로 access token / refresh token을 다루고 있다.

즉 브라우저가 인증 관련 쿠키를 자동으로 보낸다.

이 구조는 브라우저 친화적이고 편하지만, 그만큼 CSRF 방어가 꼭 필요하다.

그래서 현재 프로젝트는:

- access / refresh는 쿠키 기반 인증
- CSRF는 별도 쿠키 + 헤더 비교

조합으로 방어하고 있다.

즉 `X-CSRF-TOKEN`은 쿠키 기반 인증을 안전하게 쓰기 위한 보완 장치라고 보면 된다.

---

## 10. 한 줄 예시

예를 들어 프론트 요청은 개념적으로 이런 형태가 된다.

```http
Cookie: access_token=...; XSRF-TOKEN=abc123
X-CSRF-TOKEN: abc123
```

서버는:

- 쿠키의 `XSRF-TOKEN = abc123`
- 헤더의 `X-CSRF-TOKEN = abc123`

이 두 값이 같은지 확인한다.

같으면 통과, 다르거나 없으면 차단한다.

---

## 11. 한 줄 정리

`X-CSRF-TOKEN`은 쿠키 기반 인증에서 CSRF 공격을 막기 위해 프론트가 별도로 넣어주는 보안 헤더이고, 현재 프로젝트에서는 `XSRF-TOKEN` 쿠키 값과 이 헤더 값을 비교해서 정상 요청인지 확인한다.
