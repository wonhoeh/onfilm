## [대표 사례]

### 1. Kafka 기반 비동기 미디어 인코딩 파이프라인 구축 및 중복 callback 멱등성 처리

#### [문제]

- 사용자는 필모그래피 편집 과정에서 썸네일, 트레일러, 본편 영상 같은 대용량 미디어를 업로드해야 했습니다.
- 단순 CRUD나 동기 처리로 구현하면 업로드 후 ffmpeg 인코딩까지 API 서버가 직접 처리해야 해서 응답 지연과 서버 병목이 커질 수 있었습니다.
- 또한 인코딩은 단순 성공/실패가 아니라 `REQUESTED`, `PROCESSING`, `DONE`, `FAILED` 같은 중간 상태를 관리해야 했기 때문에 일반적인 파일 업로드 구조만으로는 부족했습니다.
- Kafka는 at-least-once 전달 방식이라 worker가 네트워크 재시도 등으로 같은 callback을 중복 전송할 수 있었습니다. 수정 전에는 이미 `DONE` 상태인 job에 중복 callback이 오면 예외가 발생해 worker가 실패로 인식하고 재시도하는 무한 반복 문제가 있었습니다.

```
[수정 전 문제 흐름]
worker → DONE callback 전송 → API 서버 처리 성공
→ 네트워크 끊김 → worker 재시도
→ API 서버: 이미 DONE인데 markDone() 호출 → 500 에러
→ worker: "또 실패, 재시도" → 무한반복
```

#### [내가 한 일]

- API 서버와 worker 서버 책임을 분리하는 비동기 구조를 설계했습니다.
- API 서버는 업로드 완료 후 Kafka로 인코딩 요청 메시지를 발행하고, `media_job` 테이블에 작업 상태를 저장하도록 구성했습니다.
- worker는 메시지를 consume 해서 ffmpeg 인코딩과 HLS 결과물 생성을 수행하고, 내부 callback API로 상태와 결과 경로를 다시 반영하도록 설계했습니다.
- `targetBucket`, `targetKey` 규칙을 정리해 dev에서는 로컬 저장소, prod에서는 S3 저장소로 해석되도록 환경별 저장 방식을 분리했습니다.
- HLS 재생까지 연결하기 위해 프론트 플레이어를 `m3u8` 기반으로 수정하고, Safari는 native HLS, 그 외 브라우저는 `hls.js`를 사용하도록 처리했습니다.
- 중복 callback 문제를 해결하기 위해 terminal 상태(DONE, FAILED)에서 동일 상태로의 중복 callback은 조용히 무시(early return)하도록 처리해 멱등성을 확보했습니다. 단, DONE → FAILED처럼 진짜 잘못된 전이는 예외를 던져 명확히 오류로 처리했습니다.

```java
public void markDone(Instant completedAt) {
    if (this.status == MediaEncodeJobStatus.DONE) {
        return; // 중복 callback 무시
    }
    if (this.status != MediaEncodeJobStatus.PROCESSING) {
        throw new IllegalStateException("INVALID_MEDIA_JOB_STATUS_TRANSITION");
    }
    ...
}
```

#### [성과]

- API 서버가 무거운 인코딩 작업을 직접 처리하지 않도록 분리해 요청-응답 경로를 단순화했습니다.
- 업로드, 인코딩, 결과 반영의 책임이 명확히 나뉘어 이후 확장과 운영 포인트를 구분하기 쉬운 구조를 만들었습니다.
- `media_job` 상태를 별도로 관리해 비동기 작업의 진행 상태를 추적할 수 있게 했고, HLS 재생까지 하나의 흐름으로 연결했습니다.
- 중복 callback에 대해 200 응답을 반환해 worker의 불필요한 재시도 루프를 차단하고 멱등성을 확보했습니다.

| 상황 | 수정 전 | 수정 후 |
|---|---|---|
| DONE → DONE (중복 callback) | 500 에러 → worker 재시도 무한반복 | 200 응답 → 재시도 없음 |
| FAILED → FAILED (중복 callback) | 500 에러 → worker 재시도 무한반복 | 200 응답 → 재시도 없음 |
| DONE → FAILED (잘못된 전이) | 500 에러 | 예외 발생 (의도된 동작) |

#### [배운 점]

- Kafka at-least-once 환경에서 비동기 처리는 메시지 발행만으로 끝나는 것이 아니라, 중복 전달 상황까지 고려한 멱등성 설계가 필요하다는 점을 배웠습니다.
- 중복 callback에 예외를 던지면 worker가 실패로 인식해 오히려 더 많은 중복 callback이 발생하는 역효과가 생긴다는 점을 직접 경험했습니다.

### 2. 비동기 도메인의 상태 머신 설계와 정합성 관리

#### [문제]

- `media_job`은 `REQUESTED → PROCESSING → DONE / FAILED`를 거치는 상태 기반 도메인이었습니다.
- worker가 네트워크 재시도나 예외 상황으로 callback을 중복 호출할 경우, 이미 완료된 작업의 상태가 덮어쓰여질 수 있었습니다.
- 상태 변경 로직이 여러 레이어에 흩어지면 어떤 상태에서 어떤 전이가 허용되는지 추론하기 어려워지고, 잘못된 전이를 방어하기도 어렵습니다.

#### [내가 한 일]

- 허용되는 상태 전이를 명확히 정의하고, 허용되지 않는 전이 시도를 차단하는 상태 가드를 서비스 레이어에 적용했습니다.
- 상태 변경 책임을 서비스 레이어에 집중시켜, 컨트롤러나 외부 callback이 직접 상태를 조작하지 않도록 구조화했습니다.
- worker callback이 중복 도달하더라도 이미 `DONE` 또는 `FAILED` 상태인 경우 무시하도록 처리해 멱등성을 확보했습니다.

#### [성과]

- 상태 전이 규칙이 코드 한 곳에 모여 있어 새로운 상태나 전이를 추가할 때 영향 범위를 파악하기 쉬워졌습니다.
- 비동기 흐름에서 중복 callback이나 순서 역전이 발생해도 상태 정합성이 유지되는 구조를 만들었습니다.

#### [배운 점]

- 주문, 결제, 배송처럼 여러 상태를 거치는 커머스 도메인에서도 동일한 설계가 필요하다는 점을 체감했습니다. 상태 전이 규칙을 명확히 정의하고 한 곳에서 통제하는 것이 도메인 정합성을 지키는 핵심이라는 걸 배웠습니다.

### 3. JWT / Refresh Token / CSRF 기반 인증·보안 구조 설계 및 개선

#### [문제]

- 서비스에 로그인, 토큰 재발급, private page 접근 제어가 필요했고, 브라우저 기반 서비스라 쿠키 기반 인증과 CSRF 방어를 함께 고려해야 했습니다.
- 특히 worker가 내부 callback API를 호출하는 과정에서 브라우저용 CSRF 정책과 서버 간 호출 정책이 충돌하는 문제가 발생했습니다.
- 또한 access token, refresh token, CSRF token 생성 책임이 컨트롤러에 몰려 있어 구조가 산만했습니다.

#### [내가 한 일]

- JWT access token과 DB 기반 refresh token rotation 구조를 적용했습니다.
- `SecurityFilterChain`, `JwtAuthFilter`, `JwtProvider`, `Authentication`, `SecurityContext` 흐름을 정리하고 인증 구조를 명확히 분리했습니다.
- CSRF 방어 필터를 두어 브라우저 요청에서는 `XSRF-TOKEN` 쿠키와 `X-CSRF-TOKEN` 헤더를 함께 검증하도록 했습니다.
- worker 내부 API 호출이 CSRF에 막히는 문제를 분석하고, dev 환경에서는 `/internal/api/**`를 예외 처리하도록 분리했습니다.
- `AuthCookieFactory`를 도입해 access / refresh / csrf 쿠키 생성 책임을 컨트롤러에서 분리했습니다.

#### [성과]

- JWT 기반 인증, refresh token rotation, CSRF 방어 흐름을 브라우저 기반 서비스에 맞게 정리할 수 있었습니다.
- worker callback 403 문제를 단순 권한 문제가 아니라 CSRF 정책 충돌 문제로 분석하고 dev/prod 정책을 분리했습니다.
- 인증 관련 책임을 `AuthService`, `AuthCookieFactory`, Security 설정으로 나눠 구조를 더 읽기 쉽게 만들었습니다.

#### [배운 점]

- 보안은 단순히 필터를 붙이는 것이 아니라, 브라우저 요청과 서버 간 요청의 성격 차이를 이해하고 정책을 나눠야 한다는 점을 배웠습니다.

### 4. 프로덕션 수준 AWS 네트워크 아키텍처 설계

#### [문제]

- EC2에 서버를 직접 올리는 구조는 API 서버, 인코딩 서버, RDS가 모두 퍼블릭 네트워크에 노출되어 외부에서 직접 접근 가능한 상태였습니다.
- 트래픽을 서버가 직접 받으면 서버 IP가 외부에 노출되고, 이후 스케일아웃 시 IP 변경에 따른 연쇄 수정이 발생하는 구조적 취약점이 있었습니다.
- 인코딩 서버는 외부 인바운드가 필요 없지만, S3 업로드나 패키지 설치를 위한 아웃바운드 인터넷 연결은 필요했습니다.

#### [내가 한 일]

- VPC 내 퍼블릭/프라이빗 서브넷을 분리하고, API 서버·인코딩 서버·RDS를 프라이빗 서브넷에 배치했습니다.
- 인바운드 트래픽은 ALB를 통해서만 API 서버로 진입하도록 구성하고, 보안 그룹으로 ALB → API 서버 간 통신만 허용했습니다.
- 프라이빗 서브넷의 아웃바운드 트래픽(S3 업로드, 패키지 설치 등)은 NAT Gateway를 통해 나가도록 라우팅 테이블을 설정했습니다.
- 서버 직접 접근이 필요한 운영·디버깅 상황을 위해 퍼블릭 서브넷에 Bastion Host를 두고 SSH 터널링으로만 접근 가능하도록 제한했습니다.

#### [성과]

- 외부에서 API 서버·인코딩 서버·RDS로의 직접 접근 경로를 차단해 공격 노출면을 줄였습니다.
- 인바운드 트래픽이 ALB에 집중되어 이후 서버 스케일아웃 시 서버 IP 변경 없이 대응 가능한 구조를 만들었습니다.
- 아웃바운드 트래픽 경로를 NAT Gateway로 통일해 프라이빗 서브넷 서버들의 외부 연결을 일관되게 관리할 수 있게 됐습니다.

#### [배운 점]

- 보안은 애플리케이션 레이어뿐 아니라 네트워크 레이어에서도 설계해야 한다는 점을 배웠습니다. 퍼블릭/프라이빗 서브넷 분리, 보안 그룹 설계, 트래픽 흐름 정의가 실무 AWS 아키텍처의 기본임을 직접 경험했습니다.

### 5. dev / prod 환경 분리와 로컬 E2E 테스트 환경 구축

#### [문제]

- 운영 환경은 S3와 worker 기반 구조를 전제로 하고 있었지만, 로컬 개발 환경에서는 동일한 흐름을 재현하기 어려웠습니다.
- 특히 presigned upload, 로컬 파일 서빙, Kafka producer-consumer 흐름, HLS 결과물 확인을 한 번에 검증하기 어려워 개발 효율이 낮았습니다.

#### [내가 한 일]

- `dev = local`, `prod = production` 기준으로 저장소 설정을 분리했습니다.
- dev 환경에서는 로컬 파일 시스템을 bucket-aware 하게 사용하도록 정리하고, `/files/**` 경로로 정적 서빙되도록 맞췄습니다.
- 로컬 raw-upload 엔드포인트를 추가해 dev에서도 업로드 → complete → Kafka 발행 흐름을 실제와 유사하게 테스트할 수 있도록 구성했습니다.
- worker가 저장하는 `targetBucket / targetKey` 규칙을 기준으로 API 서버와 저장소 경로 해석 방식을 맞췄습니다.
- 관련 동작 흐름과 내부 callback API까지 문서화해 로컬에서도 producer-kafka-consumer 테스트가 가능하도록 정리했습니다.

#### [성과]

- dev 환경에서도 운영과 유사한 미디어 처리 흐름을 재현할 수 있게 되어, 기능 검증과 디버깅 효율을 높일 수 있었습니다.
- 저장 경로, 공개 URL, worker 결과물 위치 해석 기준을 통일해 환경별 차이로 인한 오류를 줄였습니다.
- 단순 기능 구현을 넘어서 "개발 가능한 환경"을 함께 설계하는 경험을 할 수 있었습니다.

#### [배운 점]

- 기능 자체뿐 아니라 로컬에서 실제 흐름을 검증할 수 있는 환경을 만들어 두어야 개발 속도와 품질이 함께 올라간다는 점을 배웠습니다.

### 6. AI 협업 워크플로우 구축 — 시행착오를 규칙으로 축적

#### [문제]

- Claude Code, Cursor 같은 AI 도구를 사용해 코드를 생성했지만, 지침 없이 "기능을 만들어줘" 방식으로 요청하다 보니 동작은 하지만 설계 원칙이 지켜지지 않은 코드가 반복적으로 생성됐습니다.
- 예를 들어 소유권 검증 로직이 컨트롤러마다 복붙되거나, N+1 쿼리가 발생하는 구조로 코드가 생성되는 문제가 있었습니다.
- 이런 문제를 매번 발견하고 수정하는 과정에서 불필요한 반복 작업이 발생했습니다.

#### [내가 한 일]

- 문제가 발생할 때마다 단순히 수정하는 데 그치지 않고, 원인을 분석해 재발하지 않도록 규칙화했습니다.
- 접근 제어 분산 문제 → SRP 위반임을 파악 → "소유권 검증은 서비스 레이어에서만, write 메서드는 personId를 파라미터로 받지 않는다"는 규칙 도출
- N+1 문제 → p6spy로 실측 후 원인 분석 → "컬렉션은 LAZY 유지, 필요한 경우 fetch join으로 명시적 로딩"이라는 규칙 도출
- 도출된 규칙을 `CLAUDE.md`에 누적해서, 이후 AI가 코드를 생성할 때 자동으로 해당 규칙이 적용되도록 구성했습니다.

#### [성과]

- 같은 유형의 문제가 반복 생성되는 빈도가 줄었고, AI에게 매번 동일한 지침을 설명하는 반복 작업이 사라졌습니다.
- 단순 코드 생성 도구 활용이 아니라, 시행착오 → 원인 분석 → 규칙화 → 자동 적용의 피드백 루프를 만들어 AI 활용 품질을 점진적으로 높이는 워크플로우를 구축했습니다.

#### [배운 점]

- AI 도구의 효용은 프롬프트 품질에 비례합니다. "무엇을 만들어라"가 아니라 "어떤 원칙을 지켜서 만들어라"를 명시해야 의도한 설계가 유지됩니다.
- 문제를 수정하는 것과 재발을 막는 것은 다른 작업입니다. 수정에 그치지 않고 규칙으로 만들어 두는 것이 장기적인 개발 속도를 높인다는 점을 배웠습니다.

### 7. 스토리보드 조회 N+1 문제 발견 및 해결

#### [문제]

- 스토리보드 프로젝트 목록 API(`GET /api/people/{publicId}/storyboard/projects`)를 호출할 때, 프로젝트 수만큼 씬(scene) 조회 쿼리가 추가로 발생하는 N+1 문제가 있었습니다.
- `StoryboardProject`의 `scenes`가 LAZY 로딩으로 설정되어 있어, 반복문 안에서 `project.getScenes()`에 접근할 때마다 별도 쿼리가 나갔습니다.

```java
// 반복문 안에서 scenes 접근 → 프로젝트마다 쿼리 1번씩 발생
for (StoryboardProject project : person.getStoryboardProjects()) {
    project.getScenes().size(); // 쿼리 N번
}
```

#### [내가 한 일]

- p6spy를 도입해 실제 발생하는 쿼리를 측정했습니다.
- 로컬 H2 환경, 프로젝트 10개 / 씬 30개 기준으로 총 **13쿼리**가 발생하는 것을 실측으로 확인했습니다.
  - person 중복 조회 2회 + 프로젝트 목록 1회 + scenes N+1 10회
- `PersonRepository`에 fetch join 쿼리 메서드를 별도로 추가해 person, storyboard_project, storyboard_scene을 한 번의 쿼리로 조회하도록 수정했습니다.
- 1:N:N 구조에서 JOIN 시 행이 곱해져 중복이 발생하기 때문에 `@EntityGraph` 대신 JPQL에 `DISTINCT`를 직접 명시할 수 있는 fetch join을 선택했습니다.

```java
@Query("""
    SELECT DISTINCT p FROM Person p
    LEFT JOIN FETCH p.storyboardProjects sp
    LEFT JOIN FETCH sp.scenes
    WHERE p.publicId = :publicId
""")
Optional<Person> findByPublicIdWithStoryboards(@Param("publicId") String publicId);
```

#### [성과]

- p6spy 실측 기준 **13쿼리 → 1쿼리**로 개선됐습니다.
- 프로젝트가 100개일 경우 기존 구조는 103쿼리가 발생하지만, 수정 후에는 여전히 1쿼리로 유지됩니다.

| 항목 | 수정 전 | 수정 후 |
|---|---|---|
| 총 쿼리 수 (프로젝트 10개) | 13개 | **1개** |
| 총 쿼리 수 (프로젝트 100개) | 103개 | **1개** |
| scenes 조회 방식 | 프로젝트별 개별 쿼리 (LAZY) | fetch join으로 한 번에 |

#### [배운 점]

- LAZY 로딩은 JPA 기본값이지만, 반복문 안에서 컬렉션에 접근하면 N+1이 발생한다는 점을 직접 실측으로 확인했습니다.
- 문제를 추측하지 않고 p6spy로 쿼리를 실측한 뒤 수정하는 흐름이 중요하다는 점을 배웠습니다.
- 기존 `findByPublicId`를 수정하지 않고 용도에 맞는 메서드를 별도로 추가해, fetch join이 필요 없는 다른 조회에 불필요한 JOIN이 생기지 않도록 분리했습니다.

## [현재 인식하고 있는 개선 포인트]

- 접근 제어 책임이 `SecurityConfig`, 페이지 라우팅 컨트롤러, 서비스 레이어에 분산되어 있어 리팩토링 과제로 정리했습니다.
- refresh token rotation과 media job callback 구간에서 동시 요청이 들어올 경우 상태 충돌 가능성이 있어, 낙관적 락 또는 DB 유니크 제약을 활용한 중복 방어 전략을 다음 과제로 잡고 있습니다.
- 공개 조회 API와 상태 polling 구간은 Redis 캐시 적용 후보로 보고 있으며, 어떤 기준으로 캐시 대상을 선정할지 설계 중입니다.
