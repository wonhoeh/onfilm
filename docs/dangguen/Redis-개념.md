# Redis 개념

## 1. Redis란

Redis는 메모리 기반 key-value 저장소다.

주로 아래 용도로 많이 사용된다.

- 캐시
- 세션 저장
- 토큰/인증 보조 저장소
- 카운터
- 랭킹
- 분산락

디스크 기반 RDBMS보다 훨씬 빠르게 읽고 쓸 수 있어서, 응답 속도 개선이나 부하 분산에 자주 활용된다.

## 2. 왜 Redis를 쓰는가

핵심 이유는 속도와 부하 절감이다.

예를 들어 매번 DB에서 읽어야 하는 데이터를 Redis에 캐싱하면:

- 응답 속도를 줄일 수 있고
- DB 부하를 낮출 수 있다

즉 Redis는 보통 “원본 저장소를 대체”하기보다 “원본 저장소 부담을 줄이는 보조 계층”으로 많이 사용된다.

## 3. 캐시로 사용할 때 기본 개념

### 3-1. Cache Aside

가장 흔한 패턴이다.

흐름:

1. 먼저 Redis 조회
2. 값이 있으면 그대로 반환
3. 값이 없으면 DB 조회
4. 조회 결과를 Redis에 저장
5. 이후 요청은 Redis에서 응답

장점:

- 구현이 단순함
- 필요한 데이터만 캐시 가능

단점:

- 첫 요청은 cache miss
- 데이터 변경 시 캐시 무효화 전략이 필요

### 3-2. Write Through / Write Back

실무에서 일부 쓰지만, 면접에서는 Cache Aside를 가장 기본으로 알고 있으면 충분하다.

## 4. TTL

Redis 캐시는 보통 TTL(Time To Live)을 둔다.

즉:

- 일정 시간이 지나면 자동 만료

이유:

- 원본 DB와 영원히 값이 어긋나지 않게 하기 위해서
- 캐시 공간을 오래 점유하지 않게 하기 위해서

## 5. 캐시를 쓸 때 주의할 점

### 5-1. 캐시 무효화

데이터를 수정했는데 캐시를 안 지우면 오래된 값이 계속 반환될 수 있다.

즉 Redis는 “읽기 성능”을 올려주지만, 수정 시 캐시 동기화 전략까지 같이 설계해야 한다.

### 5-2. Cache Stampede

많은 요청이 동시에 들어왔는데 캐시가 한 번에 만료되면, 모두 DB로 몰리는 문제가 생길 수 있다.

대응:

- TTL 분산
- mutex
- 사전 재생성

### 5-3. 정합성 문제

캐시는 결국 원본 DB와 잠시 어긋날 수 있다.

그래서:

- 실시간 정합성이 매우 중요한 데이터
- 금전/재고/결제 핵심 데이터

는 캐시 적용 범위를 신중하게 봐야 한다.

## 6. Redis를 어디에 쓰면 좋은가

### 6-1. 조회 성능 개선

- 자주 조회되는 공개 프로필
- 인기 상품 목록
- 카테고리/기획전 목록
- 메인 화면 요약 데이터

### 6-2. 카운터

- 조회수
- 좋아요 수
- 실시간 집계

### 6-3. 세션/토큰 보조 저장소

- refresh token 블랙리스트
- access token blacklist
- 로그인 세션 메타데이터

### 6-4. 분산락

- 여러 인스턴스에서 동시에 같은 자원을 잡지 못하게 해야 할 때

## 7. 커머스 도메인에서 Redis 예시

커머스 도메인에서는 아래처럼 설명하면 좋다.

- 인기 상품/기획전 캐시
- 스토어 홈 화면 캐시
- 재고 자체보다는 재고 조회 결과 보조 캐시
- 주문/결제 직전 중복 요청 방지용 짧은 키
- 분산락이 필요한 고경합 작업

다만 재고 차감, 결제 승인 같은 정합성이 매우 중요한 핵심 로직은 Redis만 믿고 처리하기보다 DB와 함께 설계해야 한다.

## 8. 현재 프로젝트와 연결할 수 있는 부분

현재 프로젝트에는 Redis를 실제로 적용한 코드는 없다.

즉 아래는 아직 없다.

- 조회 캐시
- 토큰 캐시
- 분산락
- 실시간 카운터

대신 현재 프로젝트 기준으로 Redis를 적용한다면 아래 후보를 말할 수 있다.

### 8-1. 공개 프로필 상세 조회 캐시

- 배우 프로필
- 필모그래피
- 갤러리

이런 읽기 비중이 높은 공개 데이터는 캐싱 대상 후보가 될 수 있다.

### 8-2. 미디어 job 상태 조회 캐시

worker 상태 polling 이 많아지면 DB 대신 짧은 TTL 캐시를 둘 수 있다.

### 8-3. 토큰/세션 보조 저장소

현재 refresh token은 DB 기반인데, 향후 다중 세션 제어나 블랙리스트 전략이 더 필요해지면 Redis를 보조 저장소로 고려할 수 있다.

## 9. 면접에서 말하기 좋은 문장

`Redis는 원본 DB를 대체하는 저장소라기보다, 응답 속도를 높이고 DB 부하를 낮추기 위한 보조 계층으로 이해하고 있습니다. 그래서 자주 조회되는 공개 데이터나 인기 목록은 캐시 대상으로 보고, 결제나 재고처럼 정합성이 더 중요한 데이터는 DB 중심으로 두되 필요하면 짧은 TTL 캐시나 분산락 용도로 제한적으로 활용하는 것이 적절하다고 생각합니다. 현재 프로젝트에는 Redis를 직접 적용하진 않았지만, 공개 프로필 조회나 media job polling 같은 구간에는 적용 후보를 생각해볼 수 있습니다.`

## 10. 한 줄 요약

Redis는 메모리 기반 key-value 저장소로, 조회 성능 개선과 부하 분산에 강점이 있으며 캐시 무효화와 정합성까지 함께 고려해서 적용해야 한다.
