# 당근 커머스팀 면접 준비

## 1. 트랜잭션/동시성 질문 준비

이 문서는 당근 커머스팀 면접에서 나올 수 있는 트랜잭션/동시성 질문에 대해 예시 답변을 정리한 문서다.

답변 방향은 아래 순서를 기준으로 잡는 것이 좋다.

1. 문제 상황 설명
2. 어떤 전략을 선택할지 설명
3. 왜 그 전략을 선택하는지 설명
4. 현재 프로젝트에서 연결 가능한 사례 제시

## 2. 결제 중복 요청은 어떻게 막을 것인가

### 예시 답변

`결제는 사용자가 새로고침하거나 네트워크 재시도로 동일 요청을 여러 번 보낼 수 있기 때문에, 멱등성을 먼저 보장해야 한다고 생각합니다. 그래서 주문 생성 시 paymentRequestId 같은 멱등 키를 두고, 같은 키로 들어온 결제 요청은 DB unique constraint나 별도 idempotency 테이블로 한 번만 처리되게 막겠습니다. 그 다음 결제 상태는 CREATED, APPROVED, FAILED, CANCELED처럼 상태 전이 규칙을 두고, 이미 승인된 결제에 대해 다시 승인 로직이 타지 않도록 막겠습니다. 외부 결제사 연동까지 포함되면 애플리케이션 레벨 체크만으로는 부족하므로 DB 제약과 상태 전이를 같이 가져가는 방식이 안전하다고 봅니다.`

### 현재 프로젝트와 연결해서 말하는 방법

`현재 프로젝트에서도 완전히 같은 결제 도메인은 아니지만, 비슷한 관점으로 media_job 상태를 REQUESTED -> PROCESSING -> DONE|FAILED 로만 전이되게 막았습니다. [MediaEncodeJob.java](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/kafka/entity/MediaEncodeJob.java) 에서 잘못된 상태 변경은 INVALID_MEDIA_JOB_STATUS_TRANSITION 으로 막고 있습니다. 또 refresh token도 [RefreshTokenService.java](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/token/service/RefreshTokenService.java) 에서 rotation 시 기존 토큰을 revoke 하고 새 토큰을 발급하는 구조로 중복 사용 위험을 줄이려 했습니다. 다만 결제 수준의 멱등 키나 DB unique constraint 기반 중복 결제 방지는 아직 구현하지 않았기 때문에, 커머스 도메인에서는 그 부분을 추가하겠다고 답하는 것이 정직하다고 생각합니다.`

### 짧은 답변 버전

`결제 중복 요청은 멱등 키와 상태 전이 규칙으로 막고, 최종적으로는 DB unique constraint로 한 번 더 보장하겠습니다. 현재 프로젝트에서도 media_job 상태 전이를 제한하는 방식으로 정합성을 관리한 경험이 있습니다.`

## 3. 재고 차감에서 어떤 락 전략을 쓸 것인가

### 예시 답변

`재고 차감은 동시성 충돌이 직접 금전 문제로 이어질 수 있어서, 우선 재고 경쟁이 얼마나 심한지에 따라 전략을 나눌 것 같습니다. 일반적인 상품처럼 충돌 빈도가 높지 않으면 낙관적 락이나 조건부 update로 먼저 시도합니다. 예를 들어 stock > 0 조건으로 update 하고 영향받은 row 수가 1인지 확인하는 방식입니다. 반면 한정 수량 상품이나 인기 상품처럼 동시에 몰릴 가능성이 크면 비관적 락이나 재고 전용 직렬화 처리를 고려하겠습니다. 핵심은 무조건 강한 락부터 거는 게 아니라, 충돌 빈도와 성능 비용을 같이 보고 선택하는 것입니다.`

### 현재 프로젝트와 연결해서 말하는 방법

`현재 프로젝트에는 재고 차감처럼 고경쟁 자원이 직접 등장하는 도메인은 없어서 실제 락을 적용한 코드는 없습니다. 대신 상태 정합성은 트랜잭션과 상태 전이 가드로 먼저 관리하고 있습니다. 그래서 면접에서는 현재 프로젝트에서는 락까지 갈 정도의 고경합 도메인은 없었고, 커머스 재고라면 충돌 빈도에 따라 낙관적 락과 비관적 락을 선택하겠다고 설명하는 것이 맞다고 생각합니다.`

### 짧은 답변 버전

`충돌이 낮으면 낙관적 락이나 조건부 update, 충돌이 높고 oversell 리스크가 크면 비관적 락을 선택하겠습니다.`

## 4. 낙관적 락과 비관적 락은 언제 선택할 것인가

### 예시 답변

`낙관적 락은 충돌이 자주 나지 않는다고 가정하고, 충돌이 발생했을 때만 재시도하는 방식이라 읽기 성능과 동시성을 더 잘 살릴 수 있습니다. 그래서 프로필 수정, 순서 변경, 일반적인 백오피스 수정처럼 같은 데이터를 동시에 자주 수정하지 않는 경우에 적합합니다. 반면 비관적 락은 처음부터 다른 트랜잭션의 접근을 막기 때문에 충돌 비용이 매우 크거나, 한 번의 충돌이 치명적인 도메인에 적합합니다. 예를 들어 재고 차감, 중복 결제 방지, 좌석 선점처럼 실패 비용이 큰 경우입니다. 저는 기본적으로는 낙관적 락을 먼저 검토하고, 실제 경합이 높거나 oversell 같은 문제가 생기면 비관적 락으로 강화하는 방향이 더 현실적이라고 생각합니다.`

### 현재 프로젝트와 연결해서 말하는 방법

`현재 프로젝트에는 @Version 기반 낙관적 락이나 @Lock 기반 비관적 락을 실제로 적용한 엔티티는 아직 없습니다. 대신 [MediaEncodeJob.java](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/kafka/entity/MediaEncodeJob.java) 처럼 상태 전이 자체를 강하게 제한해서 정합성을 먼저 잡았고, [RefreshTokenService.java](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/token/service/RefreshTokenService.java) 는 트랜잭션 안에서 revoke와 재발급을 처리하도록 구성했습니다. 그래서 현재 프로젝트는 트랜잭션과 상태 전이 가드 중심이고, 이후 커머스 도메인으로 확장하면 고경합 자원에는 락 전략을 추가하겠다고 설명하는 것이 자연스럽습니다.`

### 짧은 답변 버전

`기본은 낙관적 락, 실패 비용이 큰 고경합 도메인은 비관적 락이라고 생각합니다.`

## 5. 현재 프로젝트를 과장 없이 설명하는 방법

면접에서는 없는 걸 있다고 말하면 오히려 불리하다.

현재 프로젝트 기준으로는 아래처럼 설명하는 것이 적절하다.

### 실제로 말할 수 있는 것

- `@Transactional` 기반으로 상태 변경과 저장을 한 트랜잭션 안에서 처리한 경험
- `MediaEncodeJob` 상태 전이 가드로 잘못된 상태 변경을 막은 경험
- refresh token rotation 구조로 기존 토큰 revoke 후 새 토큰을 발급한 경험
- 읽기 전용 서비스에는 `@Transactional(readOnly = true)` 를 적용한 구조

### 아직 없는 것

- `@Version` 기반 낙관적 락
- `@Lock(PESSIMISTIC_WRITE)` 기반 비관적 락
- 네임드 락
- 결제/재고 수준의 멱등 키 설계

### 그래서 답변은 이렇게 가져가는 것이 좋다

`현재 프로젝트에서는 상태 전이와 트랜잭션으로 정합성을 우선 관리했고, 재고 차감이나 중복 결제처럼 더 높은 수준의 경쟁 상황이 생기는 커머스 도메인에서는 멱등 키와 락 전략을 추가하는 식으로 확장하겠다고 생각합니다.`

## 6. 면접용 압축 답변 모음

### 결제 중복 요청

`멱등 키와 상태 전이 규칙으로 먼저 막고, 최종적으로는 DB unique constraint로 중복 요청을 차단하겠습니다. 현재 프로젝트에서도 media_job 상태 전이를 제한하는 방식으로 비슷한 정합성 문제를 관리한 경험이 있습니다.`

### 재고 차감

`충돌이 낮으면 낙관적 락이나 조건부 update, 충돌이 높고 oversell 리스크가 크면 비관적 락을 선택하겠습니다.`

### 낙관적 락 vs 비관적 락

`기본은 낙관적 락이고, 실패 비용이 크고 경쟁이 높은 도메인에서는 비관적 락을 선택하겠습니다.`

## 7. 한 줄 정리

당근 커머스팀 면접에서는 락을 많이 안다는 것보다, 어떤 도메인에서 왜 그 전략을 선택하는지 설명할 수 있는지가 더 중요하다. 현재 프로젝트 사례는 상태 전이와 트랜잭션 중심으로 연결하고, 커머스 도메인에서는 멱등 키와 락 전략으로 확장해 말하는 것이 가장 자연스럽다.
