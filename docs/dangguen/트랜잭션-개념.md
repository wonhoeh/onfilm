# 트랜잭션 개념

## 1. 트랜잭션이란

트랜잭션은 하나의 작업을 논리적으로 묶어서, 전부 성공하거나 전부 실패하게 만드는 실행 단위다.

예를 들어 주문 생성 과정에서:

- 주문 저장
- 결제 요청 저장
- 재고 차감

이 세 단계가 있다고 할 때, 중간 하나만 성공하고 나머지가 실패하면 데이터가 꼬일 수 있다.

트랜잭션은 이런 상황에서:

- 모두 성공하면 commit
- 중간에 하나라도 실패하면 rollback

되게 만드는 개념이다.

## 2. 왜 필요한가

트랜잭션이 필요한 이유는 정합성 때문이다.

예:

- 주문은 생성됐는데 결제는 실패
- 재고는 줄었는데 주문은 없음
- 토큰은 revoke 됐는데 새 토큰은 발급 안 됨

이런 중간 상태를 막기 위해 트랜잭션이 필요하다.

## 3. ACID

트랜잭션을 설명할 때 가장 기본이 되는 개념이 ACID다.

### 3-1. Atomicity

원자성

- 전부 성공하거나
- 전부 실패해야 한다

즉 작업을 쪼개서 일부만 반영되면 안 된다.

### 3-2. Consistency

일관성

트랜잭션 전후로 데이터가 비즈니스 규칙을 만족해야 한다.

예:

- 재고는 음수가 되면 안 됨
- 승인된 결제는 다시 승인되면 안 됨
- 이미 완료된 작업은 다시 처리되면 안 됨

### 3-3. Isolation

격리성

동시에 여러 트랜잭션이 실행될 때 서로 간섭하지 않도록 하는 성질이다.

이 개념이 락, 격리수준과 연결된다.

### 3-4. Durability

지속성

commit 된 결과는 시스템 장애가 나더라도 보존되어야 한다.

## 4. 격리 수준

격리 수준은 동시에 실행되는 트랜잭션 사이를 얼마나 강하게 분리할지 정하는 기준이다.

강하게 분리할수록 정합성은 좋아지지만, 성능과 동시성은 떨어질 수 있다.

### 4-1. Read Uncommitted

- 다른 트랜잭션이 아직 commit 하지 않은 값도 읽을 수 있음
- Dirty Read 가능

실무에서는 거의 쓰지 않는다.

### 4-2. Read Committed

- commit 된 데이터만 읽음
- Dirty Read는 막지만
- Non-repeatable Read는 가능

많은 DB/서비스에서 기본적으로 자주 언급되는 수준이다.

### 4-3. Repeatable Read

- 같은 트랜잭션 안에서 같은 row를 다시 읽으면 같은 값이 보장됨
- MySQL InnoDB 기본 격리 수준

실무에서 자주 쓰인다.

### 4-4. Serializable

- 가장 강한 격리 수준
- 거의 직렬 실행처럼 동작
- 정합성은 강하지만 성능 비용이 큼

고성능 서비스에서는 신중하게 선택해야 한다.

## 5. 격리 수준에서 생길 수 있는 문제

### 5-1. Dirty Read

아직 commit 되지 않은 값을 읽는 문제

### 5-2. Non-repeatable Read

같은 트랜잭션 안에서 같은 데이터를 두 번 읽었는데 값이 달라지는 문제

### 5-3. Phantom Read

같은 조건으로 조회했는데, 중간에 다른 트랜잭션이 데이터를 추가/삭제해서 결과 row 수가 달라지는 문제

## 6. InnoDB와 트랜잭션

MySQL InnoDB는 트랜잭션과 row-level lock을 지원하는 스토리지 엔진이다.

핵심적으로 기억하면 좋은 것은:

- 트랜잭션 지원
- row-level lock 지원
- MVCC 기반 읽기 처리
- 기본 격리 수준은 Repeatable Read

즉 MySQL에서 트랜잭션, 락, 동시성을 이야기할 때 InnoDB는 거의 기본 전제라고 보면 된다.

## 7. Spring에서의 트랜잭션

Spring에서는 보통 `@Transactional` 로 트랜잭션을 건다.

예:

```java
@Transactional
public void updateOrder(...) {
    ...
}
```

이 뜻은:

- 메서드 시작 시 트랜잭션 시작
- 정상 종료 시 commit
- 예외 발생 시 rollback

### 7-1. readOnly = true

```java
@Transactional(readOnly = true)
```

조회 전용 트랜잭션이다.

장점:

- 의도를 명확히 표현
- 불필요한 변경 감지를 줄이는 데 도움
- 읽기 서비스와 쓰기 서비스를 구분하기 좋음

## 8. 트랜잭션을 쓸 때 중요한 점

### 8-1. 트랜잭션은 짧게 유지

트랜잭션이 길어지면:

- 락 유지 시간이 길어지고
- 병목이 생기고
- 데드락 가능성이 커진다

### 8-2. 외부 I/O는 가능하면 트랜잭션 밖으로 분리

예:

- S3 업로드
- 외부 결제사 호출
- 외부 API 호출

이런 작업을 트랜잭션 안에 오래 두면 DB 락 점유 시간이 불필요하게 길어진다.

### 8-3. 상태 전이와 함께 생각해야 함

트랜잭션만 있다고 정합성이 다 해결되는 건 아니다.

예:

- 상태 전이 규칙
- 중복 요청 방지
- 멱등성

같은 개념도 같이 봐야 한다.

### 9-1. 실제로 설명하기 좋은 사례 1: refresh token rotation

[`RefreshTokenService.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/token/service/RefreshTokenService.java)

`rotate()` 흐름은:

1. 기존 refresh token 조회
2. 기존 토큰 `markUsed()`
3. 기존 토큰 revoke
4. 새 refresh token 생성
5. 새 토큰 저장

이 작업이 하나의 트랜잭션 안에서 처리된다.

즉:

- 기존 토큰만 revoke 되고 새 토큰이 저장되지 않거나
- 새 토큰은 생겼는데 기존 토큰 revoke 가 안 되는

중간 상태를 줄이기 위한 구조다.

### 9-2. 실제로 설명하기 좋은 사례 2: media job 상태 저장

[`MediaEncodeJobInternalService.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/kafka/service/MediaEncodeJobInternalService.java)

worker callback 으로 들어온 상태 변경을 트랜잭션 안에서 처리하고, 엔티티 쪽 상태 전이 가드와 함께 정합성을 맞추고 있다.

### 9-3. 실제로 설명하기 좋은 사례 3: 조회 전용 서비스

[`MovieReadService.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/movie/service/MovieReadService.java)
[`PersonReadService.java`](/Users/whheo/Desktop/onfilm/onfilm/src/main/java/com/onfilm/domain/movie/service/PersonReadService.java)

읽기 중심 서비스는 `@Transactional(readOnly = true)` 로 구분해두었다.

이건 읽기/쓰기 역할을 분리하고, 의도를 명확하게 보여주는 예시로 말하기 좋다.

## 10. 현재 프로젝트에서 아직 없는 것

현재 프로젝트에는 아래는 아직 직접적으로 적용돼 있지 않다.

- 격리 수준을 세밀하게 조정한 사례
- 락 전략을 선택해 적용한 사례
- `@Version` 기반 낙관적 락
- `@Lock` 기반 비관적 락

즉 면접에서는 “트랜잭션은 사용했지만, 고경합 도메인 수준의 락 전략은 아직 직접 적용하지 않았다”고 솔직히 말하는 것이 맞다.

## 11. 면접에서 말하기 좋은 문장

`현재 프로젝트에서는 트랜잭션을 통해 refresh token rotation, media job 상태 반영, 읽기/쓰기 서비스 분리를 적용해 정합성을 관리했습니다. 다만 재고 차감이나 중복 결제처럼 고경합 도메인은 없어서, 락 전략까지 직접 적용한 사례는 아직 없습니다. 그런 도메인에서는 트랜잭션만으로 부족하기 때문에 락과 멱등성까지 함께 설계해야 한다고 생각합니다.`

## 12. 한 줄 요약

트랜잭션은 하나의 작업을 전부 성공하거나 전부 실패하게 묶는 실행 단위이고, ACID와 격리 수준을 통해 정합성을 보장한다. 현재 프로젝트는 트랜잭션과 상태 전이 가드 중심으로 정합성을 관리하는 구조다.
