# 당근 커머스팀 지원 준비 가이드

## 1. 공고를 어떻게 해석하면 좋은가

이 공고는 단순히 Spring 백엔드 개발자를 뽑는 공고가 아니다.

당근 안에서 새로운 커머스 경험을 만들기 위해:

- 커머스 핵심 도메인을 설계하고
- 빠르게 실험하면서
- 확장 가능한 구조를 만들고
- 여러 서비스와 자연스럽게 연결할 수 있는 사람

을 찾는 공고에 가깝다.

즉 준비 포인트도 아래처럼 잡는 게 좋다.

- 기능 구현 경험보다 문제 해결 경험
- 단순 API 개발보다 도메인 설계 경험
- 속도만이 아니라 확장성과 정합성을 같이 고려한 경험
- 협업과 실험 중심의 개발 태도

## 2. 포트폴리오를 어떤 관점으로 정리하면 좋은가

포트폴리오에서는 “무엇을 만들었다”보다 “왜 그 구조가 필요했고 어떤 문제를 풀었는가”가 더 중요하다.

가장 좋은 구조는 아래 순서다.

1. 사용자 문제
2. 기존 방식의 한계
3. 설계한 구조
4. 구현 과정의 의사결정
5. 결과와 개선 포인트

예를 들어 아래처럼 정리하는 방식이 좋다.

- 사용자가 어떤 문제를 겪었는가
- 왜 단순 CRUD나 동기 처리로는 부족했는가
- 어떤 아키텍처로 책임을 나눴는가
- 정합성, 성능, 권한, 상태 전이를 어떻게 다뤘는가
- 이후 어떤 리팩토링 포인트를 남겼는가

## 3. 현재 프로젝트를 어떻게 연결하면 좋은가

네 프로젝트에서는 아래 경험들을 커머스팀 관점으로 연결하기 좋다.

### 3-1. 비동기 파이프라인 설계 경험

- 업로드 이후 인코딩을 동기로 처리하지 않고 Kafka + worker 구조로 분리
- API 서버와 worker 서버 책임 분리
- 상태 추적용 `media_job` 테이블 설계
- internal callback API 설계

이건 커머스 도메인에서:

- 주문 생성 후 비동기 후처리
- 결제 상태 반영
- 배송 상태 업데이트
- 외부 시스템 연동

같은 흐름으로 확장해서 말하기 좋다.

### 3-2. 상태 전이와 정합성 관리 경험

- `REQUESTED -> PROCESSING -> DONE | FAILED` 전이 가드
- 잘못된 상태 변경 방지
- 상태 조회 API와 콜백 구조 분리

이건 커머스에서는 아래와 직접 연결된다.

- 주문 상태 전이
- 결제 상태 전이
- 배송 상태 전이
- 중복 처리/재처리 방지

### 3-3. 인증/권한 설계 경험

- JWT access token + DB 기반 refresh token rotation
- SecurityFilterChain, JwtAuthFilter, CSRF 방어
- 접근 제어 책임 분산을 분석하고 리팩토링 포인트 도출

이건 커머스 도메인에서:

- 사용자/판매자 권한 분리
- 민감한 API 보호
- 결제/주문 변경 요청 보호

와 연결해 설명할 수 있다.

### 3-4. 환경 분리와 운영 고려 경험

- dev/local 과 prod 구분
- local storage 와 S3 분리
- worker 결과 저장 위치 규칙 정리
- HLS 재생 구조와 정적 파일 서빙 조정

이건 운영 환경 차이와 배포 환경을 고려한 경험으로 어필하기 좋다.

## 4. 반드시 준비해야 할 커머스 도메인 개념

커머스팀 지원 전에는 아래 개념은 반드시 설명 가능해야 한다.

### 4-1. 상품

- 상품
- 옵션
- 재고
- 가격
- 판매 상태

### 4-2. 주문

- 주문 생성
- 주문 상태
- 주문 항목
- 주문 취소

### 4-3. 결제

- 결제 승인
- 결제 실패
- 결제 취소
- 주문과 결제를 왜 분리하는가

### 4-4. 배송

- 배송 준비
- 배송 중
- 배송 완료
- 배송 상태 변경 이벤트

### 4-5. 프로모션

- 쿠폰
- 할인
- 기획전
- 가격 계산 책임

## 5. 면접에서 자주 나올 만한 질문

### 5-1. 도메인 설계

- 주문과 결제를 왜 분리해야 하나
- 주문 상태 전이는 어떻게 설계할 것인가
- 중복 주문을 어떻게 막을 것인가
- 결제 성공 전에 재고를 언제 차감할 것인가

### 5-2. 트랜잭션/동시성

- 결제 중복 요청은 어떻게 막을 것인가
- 재고 차감에서 어떤 락 전략을 쓸 것인가
- 낙관적 락과 비관적 락은 언제 선택할 것인가

### 5-3. 성능/아키텍처

- 주문 트래픽이 늘면 어디가 병목이 되는가
- Redis는 어디에 적용할 수 있는가
- Kafka는 어떤 상황에서 쓰는 게 적절한가
- 어떤 API는 동기, 어떤 API는 비동기로 나눌 것인가

### 5-4. 운영/복구

- 실패한 이벤트는 어떻게 재처리할 것인가
- 멱등성은 어떻게 보장할 것인가
- 외부 결제사 연동 실패 시 어떻게 복구할 것인가

## 6. 기술적으로 강조하면 좋은 포인트

공고에서 강조한 기술 요소는 아래다.

- Kotlin, Spring Framework
- RDBMS(MySQL)
- Redis
- Kafka
- 성능 최적화
- 확장 가능한 아키텍처

즉 준비도 아래 주제로 맞추는 게 좋다.

### 6-1. RDBMS

- 인덱스
- 정규화/비정규화 판단
- 트랜잭션
- 격리 수준
- 상태 전이와 정합성

### 6-2. Redis

- 캐시
- 세션/토큰 보조 저장
- 랭킹/카운터
- 분산락 필요 여부 판단

### 6-3. Kafka

- 비동기 이벤트 처리
- 재시도
- DLT
- consumer 실패 복구
- 멱등성

### 6-4. 성능 최적화

- N+1 문제
- JPA 조회 전략
- 느린 쿼리 분석
- 병목 구간 측정
- k6 같은 부하 테스트 도구 활용

## 7. 당근 커머스팀 맞춤 어필 포인트

이 팀은 “이미 정리된 플레이북을 수행하는 사람”보다 “문제를 구조화하고 빠르게 실험하는 사람”에 가깝다.

그래서 아래처럼 말하는 게 좋다.

- 현재 단계에 맞는 구조를 선택했다
- 완벽한 설계보다 확장 가능한 최소 구조를 먼저 만들었다
- 빠르게 구현하되, 이후 리팩토링 포인트를 문서화했다
- 고객 경험과 시스템 구조를 같이 보려고 했다
- 도메인 흐름과 기술 구조를 함께 설명할 수 있다

## 8. 지원 동기는 어떻게 준비하면 좋은가

지원 동기는 기술만 이야기하면 약하다.

아래 3가지를 같이 담는 게 좋다.

- 왜 당근인가
- 왜 커머스인가
- 왜 지금 내 경험과 맞는가

### 주의할 점

당근페이를 통한 에스크로/플랫폼 결제는 당근이 이미 만들고 있는 방향이다. "이렇게 하면 좋겠다"는 제안처럼 말하면 현재 서비스를 모르는 것처럼 보여 역효과가 날 수 있다. "이 방향을 실제로 구축하는 초기 단계에 참여하고 싶다"는 식으로 바꾸는 게 좋다.

또한 "왜 커머스가 필요한가"에서 끝내지 말고, "왜 내가 이 일을 하고 싶은가(엔지니어 관점)"까지 연결해야 한다.

### 예시 답변

> 당근 사용자로서 원하는 물건이 올라와도 직접 만나야 한다는 번거로움에 거래까지 이어지지 않는 경험을 자주 했습니다. 당근이 커머스를 만들면 단순히 쇼핑 기능을 추가하는 게 아니라, 이미 신뢰 기반이 형성된 동네 커뮤니티 위에서 자연스러운 구매 흐름이 만들어진다는 점이 다른 이커머스와 다르다고 생각했습니다. 저는 비동기 처리, 상태 기반 도메인 설계, 인증 구조를 직접 설계해본 경험이 있는데, 이 경험이 주문·결제·배송이라는 커머스 핵심 도메인을 처음부터 설계하는 환경에 잘 맞을 것 같아 지원했습니다.

## 9. 실제 준비 체크리스트

### 9-1. 포트폴리오

- 프로젝트 1개를 3분 안에 설명할 수 있게 정리
- 문제 해결 사례 5개 정리 완료 (Kafka 파이프라인 / 상태 머신 / 인증·보안 / AWS 아키텍처 / 환경 분리)
- 성능/트랜잭션/권한/비동기 처리 사례 정리

### 9-2. 면접 답변

- RDB 답변 1분 버전
- Kafka 답변 1분 버전
- JWT/Security 답변 1분 버전
- 커머스 도메인 질문 답변 1분 버전

### 9-3. 문서/프로젝트 정리

- ERD
- 주요 API 흐름
- 접근 제어 구조
- 비동기 처리 흐름
- 리팩토링 포인트

## 10. 최종 전략

당근 커머스팀 준비는 아래 방향으로 잡는 게 좋다.

- “백엔드 기능 구현자”가 아니라 “도메인과 시스템을 같이 설계한 사람”으로 보이기
- 고객 경험과 시스템 구조를 연결해서 설명하기
- 커머스 기본 도메인을 말할 수 있게 준비하기
- 속도와 확장성 사이의 균형을 고민한 경험을 보여주기
- 현재 프로젝트를 커머스 플랫폼 설계 경험으로 번역해서 설명하기

## 11. 한 줄 요약

당근 커머스팀 지원 준비는 기능 나열보다 문제 해결, 도메인 설계, 비동기 처리, 상태 관리, 확장 가능한 구조 설계 경험을 중심으로 정리하는 것이 가장 효과적이다.
