# 락 개념

## 1. 락이 왜 필요한가

락은 여러 요청이 동시에 같은 데이터를 수정할 때 정합성을 지키기 위한 제어 장치다.

예를 들어 재고가 1개 남은 상품에 대해 두 결제 요청이 동시에 들어오면:

- 두 요청 모두 재고가 있다고 판단하고
- 둘 다 차감 로직을 실행해서
- 실제보다 더 많이 판매되는 문제가 생길 수 있다

이런 문제를 막기 위해 락이나 동시성 제어가 필요하다.

## 2. 기본 개념

### 2-1. 공유 락

- 읽기는 허용
- 쓰기는 막음

보통 조회 중인 데이터를 다른 트랜잭션이 함부로 수정하지 못하게 할 때 사용된다.

### 2-2. 배타 락

- 다른 읽기/쓰기 또는 쓰기를 제한
- 수정 시 주로 사용

실무에서는 “수정 시 배타 락” 개념으로 이해하면 된다.

## 3. 낙관적 락

낙관적 락은 충돌이 자주 일어나지 않는다고 가정하고, 저장 시점에만 충돌 여부를 확인하는 방식이다.

보통 JPA에서는 `@Version` 으로 구현한다.

### 동작 방식

1. 데이터를 읽음
2. 수정
3. 저장 시 version 값이 처음 읽은 값과 같은지 확인
4. 이미 다른 트랜잭션이 수정했다면 실패

### 장점

- 락을 오래 잡지 않음
- 동시성이 좋음
- 읽기 성능에 유리함

### 단점

- 충돌 시 재시도 필요
- 경쟁이 심하면 실패가 자주 날 수 있음

### 적합한 경우

- 프로필 수정
- 순서 변경
- 일반적인 백오피스 수정
- 동시에 자주 수정되지 않는 데이터

## 4. 비관적 락

비관적 락은 충돌이 날 가능성이 높다고 보고, 처음부터 DB 락을 잡고 시작하는 방식이다.

보통:

- `SELECT ... FOR UPDATE`
- JPA `PESSIMISTIC_WRITE`

같은 방식으로 구현한다.

### 동작 방식

1. 데이터를 읽는 시점에 락을 잡음
2. 다른 트랜잭션이 동시에 수정하지 못하게 막음
3. 작업이 끝나면 락 해제

### 장점

- 충돌 가능성이 높은 상황에서 안정적
- 정합성을 강하게 보장할 수 있음

### 단점

- 대기 시간이 길어질 수 있음
- 병목이 생길 수 있음
- 처리량이 떨어질 수 있음

### 적합한 경우

- 재고 차감
- 중복 결제 방지
- 좌석 선점
- 실패 비용이 큰 자원

## 5. 네임드 락

네임드 락은 특정 row가 아니라 이름 기준으로 잡는 락이다.

예:

- `product:123`
- `order:456`

MySQL의 `GET_LOCK()` 같은 방식이 대표적이다.

### 장점

- 특정 테이블 row에 직접 묶이지 않음
- 여러 테이블을 걸친 작업도 하나의 리소스 단위로 제어 가능

### 단점

- 운영 난이도가 더 높음
- 락 해제 관리가 중요함
- 잘못 쓰면 디버깅이 어려움

### 적합한 경우

- 여러 자원을 묶어서 직렬화해야 할 때
- 특정 리소스 단위로 강하게 제어하고 싶을 때

## 6. 조건부 update

이건 엄밀히 락 자체는 아니지만, 락 대신 많이 쓰이는 방식이다.

예:

```sql
update stock
set quantity = quantity - 1
where product_id = 1
  and quantity > 0
```

이후 영향받은 row 수가:

- 1이면 성공
- 0이면 실패

### 장점

- 구현이 단순함
- 락보다 가벼울 수 있음
- 재고 차감 같은 단순 원자 연산에 유용함

### 적합한 경우

- 단순 수량 차감
- oversell 방지용 원자적 감소 연산

## 7. 데드락

데드락은 서로 다른 트랜잭션이 서로가 가진 락이 풀리기를 기다리면서 멈추는 상황이다.

예:

- 트랜잭션 A: 상품 -> 주문 순으로 락 획득
- 트랜잭션 B: 주문 -> 상품 순으로 락 획득

이런 경우 서로 상대가 락을 풀기를 기다리며 데드락이 생길 수 있다.

### 예방 방법

- 항상 같은 순서로 락 잡기
- 락 범위를 최소화하기
- 트랜잭션을 짧게 유지하기

## 8. 언제 어떤 전략을 쓰는가

정리하면 보통 이렇게 선택한다.

### 충돌이 적다

- 낙관적 락

### 충돌이 많고 실패 비용이 크다

- 비관적 락

### 단순 수량 차감이다

- 조건부 update 우선 검토

### 여러 자원이나 여러 테이블을 직렬화해야 한다

- 네임드 락 검토

## 9. 면접에서 말할 수 있는 기준 문장

`기본적으로는 충돌이 적으면 낙관적 락을 먼저 검토하고, 충돌 빈도가 높고 실패 비용이 큰 도메인에서는 비관적 락을 선택합니다. 단순 재고 차감처럼 원자적 감소 연산이면 조건부 update를 먼저 고려하고, 여러 자원을 묶어서 직렬화해야 하면 네임드 락도 검토할 수 있습니다.`

## 10. 현재 프로젝트 기준으로 연결할 수 있는 부분

현재 프로젝트에는:

- `@Version` 기반 낙관적 락
- `@Lock(PESSIMISTIC_WRITE)` 기반 비관적 락
- 네임드 락

은 아직 없다.

대신 아래는 실제로 적용되어 있다.

- `@Transactional` 기반 상태 변경 처리
- `MediaEncodeJob` 상태 전이 가드
- refresh token revoke / rotation 구조

즉 현재 프로젝트는 락보다는 트랜잭션과 상태 전이 가드로 정합성을 관리하는 구조에 가깝다.

따라서 면접에서는 이렇게 연결하면 자연스럽다.

`현재 프로젝트에서는 락을 직접 적용할 정도의 고경합 도메인은 없어서 트랜잭션과 상태 전이 가드 중심으로 정합성을 관리했습니다. 다만 커머스처럼 재고나 결제처럼 고경합 자원이 생기면 낙관적 락, 비관적 락, 조건부 update를 상황에 맞게 선택할 수 있다고 생각합니다.`

## 11. 한 줄 요약

락은 동시에 같은 데이터를 수정할 때 정합성을 지키기 위한 장치이고, 충돌이 적으면 낙관적 락, 충돌이 많고 실패 비용이 크면 비관적 락을 선택하는 것이 기본이다.
