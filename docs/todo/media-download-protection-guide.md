# 미디어 다운로드 방지 검토 문서

## 결론 요약

브라우저에서 재생되는 영상은 클라이언트로 실제 데이터가 전달되기 때문에, 우클릭 차단이나 F12 차단만으로 다운로드를 완전히 막을 수는 없다.  
즉 `우클릭 저장 방지`, `DevTools 네트워크 탭에서 ts 세그먼트 다운로드 방지`는 프론트엔드만으로는 완전한 해결이 불가능하다.

현실적인 목표는 다음과 같다.

- 직접 URL 접근 난이도 높이기
- 링크 공유 재사용 방지
- 원본 파일 노출 차단
- 무단 수집 비용 증가

---

## 1. 왜 완전 차단이 어려운가

### 1-1. 우클릭 차단은 보안 기능이 아니다

프론트에서 아래 정도는 가능하다.

- `contextmenu` 이벤트 차단
- 기본 비디오 컨트롤 숨김
- 다운로드 버튼 숨김
- 드래그 방지

하지만 사용자는 여전히 아래 방식으로 파일에 접근할 수 있다.

- 브라우저 개발자도구
- 네트워크 탭
- 브라우저 확장 프로그램
- 캐시/임시파일 추출
- 화면 녹화

즉 우클릭 차단은 단순한 UX 억제 수단이지, 실제 보안 수단은 아니다.

### 1-2. HLS는 재생을 위해 세그먼트를 내려줘야 한다

HLS 재생은 보통 아래 파일을 순차적으로 받는다.

- `index.m3u8`
- `segment_000.ts`
- `segment_001.ts`
- ...

브라우저가 영상을 재생하려면 이 세그먼트 파일을 실제로 내려받아야 하므로, 사용자는 그 요청을 관찰하거나 저장할 수 있다.  
즉 `재생은 되게 하면서 네트워크 탭 다운로드는 절대 못 하게` 하는 것은 일반적인 공개 HLS 구조에서는 불가능하다.

---

## 2. 현재 프로젝트 기준으로 가능한 보호 방향

현재 프로젝트는 다음 구조에 가깝다.

- `dev`: 로컬 파일 시스템 + `/files/**` 정적 서빙
- `prod`: S3 기반 저장소
- HLS 결과물은 `targetKey` 기준으로 저장
- 플레이어는 `m3u8`를 직접 재생

이 구조에서 가장 큰 문제는 `정적 공개 URL`이다.

- `/files/{key}`
- 또는 공개 S3 URL

이 방식은 URL만 알면 누구나 접근하기 쉬워서 보호 수준이 약하다.

---

## 3. 추천 대응책

## 3-1. 1순위: 공개 정적 URL 대신 서명 기반 접근으로 변경

### 개념

영상 파일과 HLS 세그먼트를 public으로 직접 노출하지 않고, 짧은 만료 시간을 가진 signed URL 또는 signed cookie로만 접근하게 만든다.

### 적용 방식 예시

- S3 presigned GET URL
- CloudFront signed URL
- CloudFront signed cookie
- API 서버가 짧은 만료 토큰을 발급하고, 프록시 서버가 토큰 검증 후 파일 전달

### 장점

- 링크 복붙 재사용을 줄일 수 있음
- 만료 시간이 지나면 URL 재사용이 어려움
- 공개 버킷보다 훨씬 안전함

### 단점

- 구현 복잡도 증가
- HLS는 `m3u8`뿐 아니라 세그먼트 URL도 함께 고려해야 함
- 플레이어/스토리지/CDN 설계가 함께 바뀔 수 있음

### 현재 프로젝트에서의 적합성

가장 현실적인 운영 대응책이다.  
`prod`에서 `/files/**` 공개 대신 signed URL 또는 media proxy로 바꾸는 것이 우선순위가 높다.

---

## 3-2. 2순위: HLS 암호화 적용

### 개념

HLS 세그먼트 자체를 암호화하고, 플레이어는 별도 키를 받아 복호화하며 재생한다.

대표 방식:

- AES-128 HLS encryption
- SAMPLE-AES

### 장점

- 세그먼트 파일을 바로 내려받아도 즉시 재생/재사용하기 어려움
- 단순 정적 HLS보다 보호 수준이 높음

### 단점

- 키 관리가 필요함
- 플레이어/인코딩 워커/배포 구조가 복잡해짐
- DRM 수준의 완전 보호는 아님

### 현재 프로젝트에서의 적합성

워커에서 ffmpeg 인코딩을 이미 수행하므로, 장기적으로 붙일 수 있는 방향이다.  
다만 현재 단계에서는 signed URL보다 먼저 적용할 우선순위는 낮다.

---

## 3-3. 3순위: DRM 도입

### 개념

Widevine, FairPlay, PlayReady 같은 DRM을 통해 브라우저/OS 수준의 보호를 적용한다.

### 장점

- 가장 강한 보호 수준
- 상용 OTT 수준의 미디어 보호 가능

### 단점

- 구현/운영 비용이 매우 큼
- 라이선스 서버, 플레이어, 인코딩 파이프라인 모두 영향
- 현재 프로젝트 규모에서는 과할 수 있음

### 현재 프로젝트에서의 적합성

현 시점에서는 과한 선택일 가능성이 높다.  
기업용 상용 서비스나 고가 콘텐츠가 아니라면 우선 검토 대상은 아니다.

---

## 3-4. 4순위: 워터마크 적용

### 개념

영상 위에 사용자 식별 정보나 서비스 워터마크를 삽입한다.

### 장점

- 유출 억제 효과가 큼
- 법적/운영적 대응 근거가 생김

### 단점

- 다운로드 자체를 막지는 못함
- 사용자 경험에 영향을 줄 수 있음

### 현재 프로젝트에서의 적합성

완전 차단이 어려운 상황에서 실무적으로 꽤 유효한 보조 수단이다.

---

## 3-5. 5순위: 프론트 차단은 보조 수단으로만 사용

적용 가능한 예:

- 우클릭 차단
- 다운로드 메뉴 숨김
- DevTools 탐지 시 경고
- 드래그 방지

### 장점

- 구현이 쉬움
- 일반 사용자 억제 효과는 약간 있음

### 단점

- 기술적으로 쉽게 우회 가능
- 보안 대책이라고 말하기 어려움

### 현재 프로젝트에서의 적합성

운영 보안의 핵심으로 쓰면 안 되고, 보조 UX 억제 수단으로만 보는 것이 맞다.

---

## 4. 현재 프로젝트에 권장하는 실제 적용 순서

## 4-1. dev / prod를 다르게 본다

`dev`는 로컬 테스트 목적이므로 지금처럼 비교적 단순하게 공개 서빙해도 된다.  
보호가 중요한 것은 `prod`이다.

### dev

- 현재 로컬 `/files/**` 유지 가능
- 보안보다 개발 편의성 우선

### prod

- 공개 정적 URL 지양
- private 저장소 + signed URL 또는 media proxy 적용

---

## 4-2. 추천 1단계

`prod`에서 HLS 결과물을 public URL로 직접 노출하지 않고, signed URL 또는 CDN 서명 방식으로 전환한다.

예시:

- `movie/{movieId}/file/{uuid}/index.m3u8`
- `movie/{movieId}/file/{uuid}/segment_000.ts`

이 경로를 직접 공개하지 않고, 짧은 만료 시간을 가진 URL로만 접근하게 한다.

---

## 4-3. 추천 2단계

원본 파일(`raw`)은 절대 외부에서 접근 가능하게 노출하지 않는다.

현재 기준으로 중요한 원칙:

- raw source는 워커만 읽는다
- 서비스 사용자는 raw에 직접 접근하지 못한다
- 외부에 노출하는 것은 최종 결과물만

---

## 4-4. 추천 3단계

필요하면 HLS 암호화를 추가 검토한다.

적용 시 검토 대상:

- 워커 ffmpeg 인코딩 옵션
- 키 저장/전달 방식
- 플레이어 호환성

---

## 5. 포트폴리오/면접에서 어떻게 말할 수 있는가

아래처럼 설명하면 좋다.

`브라우저에서 재생되는 미디어는 클라이언트로 실제 데이터가 전달되기 때문에, 우클릭 금지나 F12 차단만으로는 다운로드를 완전히 막을 수 없다고 판단했습니다. 그래서 프론트 차단보다는 운영 환경에서 정적 공개 URL을 없애고, signed URL 또는 media proxy 기반으로 접근을 제어하는 방향이 더 현실적이라고 봤습니다. 필요 시 HLS 암호화나 워터마크 같은 보조 수단도 함께 검토할 수 있습니다.`

---

## 6. 이 프로젝트 기준 권장 결론

가장 현실적인 선택은 아래 조합이다.

1. `dev`는 현재 방식 유지
2. `prod`는 `/files/**` 직접 공개 대신 signed URL 또는 media proxy 적용
3. raw 원본은 절대 외부 공개 금지
4. 필요 시 HLS 암호화 검토
5. 우클릭 차단은 보조 수단으로만 사용

즉, “완전 다운로드 차단”을 목표로 하기보다 “직접 접근과 재사용을 어렵게 만드는 구조”로 가는 것이 맞다.
